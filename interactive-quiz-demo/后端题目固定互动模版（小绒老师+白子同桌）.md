### 4. Mysql 为什么用 B+ 树而不是跳表

**小绒老师 → 知识讲解**
> 跳表也是一种支持范围查询的数据结构，Redis 的 ZSet 就是用它实现的。但 Mysql 的数据存储在磁盘上，B+ 树的一个节点对应一个磁盘页（默认 16KB），一次 I/O 能读大量数据。而跳表的节点分散，对磁盘不友好。B+ 树的高度一般不超过 3-4 层，所以大部分查询只需要 3-4 次磁盘 I/O。

### 8. 为什么推荐自增整型做主键而不是 UUID

**小绒老师 → 知识讲解**
> 主键的选择直接影响 B+ 树的写入性能。自增整型是顺序写入的，新数据只追加到 B+ 树的右侧，页分裂少。UUID 是完全随机的，写入时可能插入到 B+ 树的中间位置，频繁触发页分裂和页调整。另外 UUID 占用 36 个字符，而 INT 只需要 4 个字节，BIGINT 也才 8 个字节——每个非主键索引都会存储主键值，UUID 会让所有索引都变得更大。

### 12. Mysql 的三个日志（binlog / redo log / undo log）

**小绒老师 → 知识讲解**

### 13. 什么是多版本并发控制（MVCC）

**小绒老师 → 知识讲解**
> MVCC（Multi-Version Concurrency Control）是 InnoDB 实现高并发读的核心机制。它通过 undo log 保留数据的历史版本，配合 ReadView（读视图）来判断事务应该看到哪个版本的数据。每行数据有两个隐藏列：DB_TRX_ID（最后修改的事务 ID）和 DB_ROLL_PTR（指向 undo log 的指针）。事务启动时生成 ReadView，通过比较事务 ID 和数据版本决定可见性。这样就做到了"读不加锁，写不阻塞读"。

### 15. 为什么要分库分表

**小绒老师 → 知识讲解**
> 分库分表是为解决单库单表性能瓶颈。分库（垂直拆分）是把不同的业务表拆分到不同数据库中，缓解单一数据库的连接数和 I/O 压力。分表（水平拆分）是把同一张表按某个键（如用户 ID）拆到多张结构相同的表中，单表数据量减少，查询和维护都更快。常见的分片策略有 range 分片（按范围）、hash 分片（按哈希值）。分库分表后还会引入跨库查询、分布式事务、全局主键等新问题。

### 17. 数据库三大范式

**小绒老师 → 知识讲解**
> 数据库设计的三大范式是规范表结构的标准。第一范式（1NF）要求每列都是不可分割的原子值。第二范式（2NF）在 1NF 基础上要求非主键列完全依赖于主键，不能部分依赖。第三范式（3NF）在 2NF 基础上要求非主键列直接依赖于主键，不能传递依赖。实际项目中不一定要严格遵循三大范式，适当的冗余可以减少关联查询，提高查询效率，这就是"反范式化"设计。

### 21. 索引是不是建得越多越好

**小绒老师 → 知识讲解**
> 索引不是越多越好。每个索引都是 B+ 树，写数据时所有索引都要同步更新。一张表上索引太多，插入、更新、删除操作会变慢。索引还占用额外的磁盘空间。另外查询优化器如果可选索引太多，也可能选错索引导致性能反而下降。一般来说单表索引控制在 5 个左右比较合理，核心原则是"只为常用的查询路径建索引"。

### 29. Mysql 的主从复制

**小绒老师 → 知识讲解**

> SpringBoot 自动装配的核心是 @EnableAutoConfiguration 注解。它通过 @Import(AutoConfigurationImportSelector.class) 加载 META-INF/spring.factories 文件中配置的所有自动配置类。每个自动配置类上都有一组 @Conditional 条件注解，比如 @ConditionalOnClass（类存在时才生效）、@ConditionalOnMissingBean（没有自定义 Bean 时才生效）。只有条件满足的配置才会生效，所以自动配置不会覆盖用户的自定义配置。

### 45. 怎么自己写一个 Starter

### 48. #{} 和 ${} 的区别

**小绒老师 → 知识讲解**
> #{} 使用预编译机制，参数用 ? 占位符，由 PreparedStatement 设置参数值，可以防止 SQL 注入。${} 是直接字符串替换，把参数值拼接到 SQL 中，存在 SQL 注入风险。#{} 还能自动给字符串类型加引号，${} 不会。一般情况下都要用 #{}，只有在表名、列名等数据库对象名动态传入时才用 ${}，并且这样的场景要做好白名单校验。

### 49. ResultMap 和 ResultType 的区别

**小绒老师 → 知识讲解**
> ResultType 直接指定返回的实体类类型，要求 SQL 查出的列名和实体类的属性名完全一致（或开启驼峰映射）。ResultMap 是自定义映射规则，可以手动配置列名和属性名的对应关系，适合字段名不对齐、多表关联查询需要返回非实体类结果的场景。结果映射比较复杂时（如一对一、一对多嵌套映射）必须用 ResultMap。

### 50. 动态 SQL 是什么

**小绒老师 → 知识讲解**
> Mybatis 动态 SQL 通过 XML 标签实现条件拼接，核心标签有：<if> 判断条件是否成立，成立则拼接 SQL 片段；<where> 自动处理 AND 和 OR 前缀，避免多余关键字的错误；<set> 在 UPDATE 中动态包含需要更新的列；<foreach> 遍历集合生成 IN 条件；<choose>/<when>/<otherwise> 实现 switch-case 逻辑。动态 SQL 的常见坑：<if> 判断字符串时要注意空字符串和 NULL 的区别；<where> 标签中第一个条件前不要加 AND。

**小绒老师 → 课堂提问**
