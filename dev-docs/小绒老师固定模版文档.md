# 小绒老师 — 按题目分类的固定互动模版

> 依据：`AI角色业务提示词文档.md` 第 2.1、3-6 节
> 性格锚点：温和专业、有条理、先鼓励再纠错、不说废话、答错换简单说法

---

## 设计原则

- **每题一套固定模版**，异步预生成，运行时零 AI 调用
- **按分类差异化语气**：数据库类偏严谨、Java 核心类偏结构化、框架类偏实战、中间件类偏原理
- **每类场景严格控制字数**，只传必要字段
- **所有模版先调用 COMMON_STUDY_ROLE_RULE + PERSONA_TEACHER_XIAORONG**

---

## 分类体系对照

| 分类 | 包含专题 | 小绒老师风格基调 |
|------|---------|----------------|
| **数据库** | Mysql | 严谨、重原理、先概念后应用 |
| **Java 核心** | Java基础、Java集合、多线程、锁、内存 | 条理分明、重底层机制 |
| **主流框架** | Spring、SpringBoot、Mybatis、SpringCloud | 实战导向、重使用场景 |
| **中间件** | Redis、RabbitMq | 原理+实践结合、重对比 |
| **前端** | HTML、CSS | 轻快、示范感强 |

## 模版清单

### 1. 课堂提问包装模版

每道题从题库取出后，小绒老师用一句"引子"包装后提问。不同分类的引子语气不同。

#### 1.1 数据库类（Mysql）

```
固定前缀：
  我们先看一道数据库的高频考题——

模版：
  {topicName}
  提示：可以从{keyHint1}和{keyHint2}两个角度回答。

示例：
  我们先看一道数据库的高频考题——
  事务的四大特性 ACID。
  提示：可以从"每个特性解决了什么问题"和"它们之间的关联"两个角度回答。

字数：≤ 40 字（不含提示）
```

#### 1.2 Java 核心类（基础 / 集合 / 多线程 / 锁 / 内存）

```
固定前缀：
  来，看一道 Java 核心知识题——

模版：
  {topicName}
  注意说清楚{keyHint1}和{keyHint2}的关系。

示例：
  来，看一道 Java 核心知识题——
  HashMap 的 put 流程。
  注意说清楚哈希冲突和树化的条件。

字数：≤ 35 字（不含提示）
```

#### 1.3 主流框架类（Spring / SpringBoot / Mybatis / SpringCloud）

```
固定前缀：
  实际项目中经常遇到的框架题——

模版：
  {topicName}
  从{keyHint1}和{keyHint2}展开说说。

示例：
  实际项目中经常遇到的框架题——
  Spring Bean 的生命周期。
  从初始化阶段和销毁阶段展开说说。

字数：≤ 35 字（不含提示）
```

#### 1.4 中间件类（Redis / RabbitMq）

```
固定前缀：
  来一道中间件原理题——

模版：
  {topicName}
  重点讲{keyHint1}，顺便提一下{keyHint2}。

示例：
  来一道中间件原理题——
  Redis 的过期删除策略。
  重点讲惰性删除和定期删除的区别，顺便提一下内存淘汰机制。

字数：≤ 40 字（不含提示）
```

#### 1.5 前端类（HTML / CSS）

```
固定前缀：
  看一道前端基础题——

模版：
  {topicName}
  说说{keyHint1}，可以配合一个例子。

示例：
  看一道前端基础题——
  CSS 盒模型。
  说说 content-box 和 border-box 的区别，可以配合一个例子。

字数：≤ 30 字（不含提示）

### 2. 知识点讲解模版

用户点击"讲讲这题"时，小绒老师讲解。先给结论再解释，不同分类重音不同。

#### 2.1 数据库类

```
模版：
  核心结论：{oneSentenceCore}
  为什么：{whyExplanation}
  {analogy}（类比）
  面试建议：{interviewTip}

示例：
  核心结论：事务的原子性就是"要么全做，要么全不做"。
  为什么：因为 InnoDB 通过 undo log 记录操作前的数据状态，如果事务失败就按 undo log 回滚。
  你可以想象成下笔前先拍照，写错了就恢复照片。
  面试建议：说完原子性可以补一句"InnoDB 的 undo log 是保证原子性的关键机制"。

字数：≤ 200 字
```

#### 2.2 Java 核心类

```
模版：
  本质：{essence}
  底层机制：{mechanism}
  常见坑：{commonPitfall}
  面试表达建议：{interviewTip}

示例：
  本质：HashMap 就是数组+链表+红黑树。
  底层机制：put 时先 hash 定位桶位置，冲突时尾插法挂链表，链表长度≥8 且数组≥64 转红黑树。
  常见坑：多线程下扩容可能死循环（JDK7），用 ConcurrentHashMap 替代。
  面试表达建议：可以说"HashMap 的核心是通过哈希算法将 key 映射到数组下标，冲突时用链地址法解决"。

字数：≤ 220 字
```

#### 2.3 主流框架类

```
模版：
  一句话概括：{summary}
  关键流程：{keyProcess}
  实际项目中注意：{projectNote}

示例：
  一句话概括：Spring Bean 生命周期就是"实例化→属性赋值→初始化→销毁"四个阶段。
  关键流程：实例化→依赖注入→setBeanName→BeanPostProcessor前置→init-method→BeanPostProcessor后置→就绪。
  实际项目中注意：可以利用 BeanPostProcessor 做自定义增强，比如记录 Bean 初始化耗时。

字数：≤ 180 字
```

#### 2.4 中间件类

```
模版：
  核心原理：{corePrinciple}
  为什么要这样设计：{designReason}
  同类对比：{comparison}
  面试表达建议：{interviewTip}

示例：
  核心原理：Redis 过期删除有定时删除、惰性删除、定期删除三种策略。
  为什么要这样设计：定时删除太耗 CPU，惰性删除可能堆积过期数据，定期删除是折中方案。
  同类对比：memcached 只有惰性删除，没有定期扫描机制。
  面试表达建议：可以说"Redis 实际用的是定期删除+惰性删除的组合策略，兼顾 CPU 和内存"。

字数：≤ 220 字
```

#### 2.5 前端类

```
模版：
  概念：{concept}
  效果展示：{effectDescription}
  使用注意：{usageNote}

示例：
  概念：CSS 盒模型中 content-box 的 width 只包含内容区，border-box 包含 content+padding+border。
  效果展示：设置 width:100px + padding:10px，content-box 实际占 120px，border-box 仍是 100px。
  使用注意：全局设置 box-sizing: border-box 可以减少布局计算的心智负担。

字数：≤ 150 字
```

### 3. 作答评估模版

用户提交答案后，小绒老师结构化评估。输出 JSON，不同分类的评分侧重点不同。

#### 3.1 数据库类 — 侧重概念准确性

```
评分重点：
  - 概念定义是否精确（权重 40%）
  - 应用场景理解（权重 30%）
  - 底层原理深度（权重 30%）

输出 JSON：
  {
    "score": 0-100,
    "level": "优秀|良好|一般|较差",
    "hitKeywords": [],
    "missKeywords": [],
    "teacherReply": "（数据库风格：先肯定准确的点，再补原理层面的遗漏）",
    "problem": "",
    "suggestion": "（建议补一句底层机制相关的表达）"
  }
```

#### 3.2 Java 核心类 — 侧重机制完整性

```
评分重点：
  - 流程完整性（权重 40%）

    "problem": "",

#### 3.3 主流框架类 — 侧重实战应用

  - 核心流程掌握（权重 30%）

输出 JSON：

    "score": 0-100,

    "missKeywords": [],

## Token 节省汇总

| 模版类型 | 每次调用传入字段数 | 输出字数上限 | 运行时是否调 AI |
|---------|-----------------|------------|---------------|
| 课堂提问包装 | 4（topicName, keyHint1, keyHint2, category） | ≤40字 | 否（预生成） |
| 知识点讲解 | 5（topicName, essence, mechanism, tip, analogy） | ≤220字 | 否（预生成） |
| 作答评估 | 6（topicName, answerSummary, keywords, userAnswer, category） | JSON 定长 | 仅低置信度时 |
| 纠错 | 4（topicName, wrongPoint, simpleExplanation, category） | ≤160字 | 否（预生成） |

小绒老师在普通刷题流程中，**仅作答评估环节在低置信度时实时调 AI**，其余全部预生成。
