# AI 教师互动激励方案评估

## 1. 背景

当前 demo 的方向是：AI 从学习开始就介入，让用户先选择学习方向，然后由 AI 老师讲知识点，讲完后布置作业，用户提交一题，AI 讲解一题。

你提出的新目标是进一步提高同学的积极学习性，让用户不只是被动听课和答题，而是在学习过程中持续被提问、被反馈、被鼓励。

同时，之前已经讨论过一个关键架构问题：

固定题库不适合每次都让 AI 实时生成整节课，否则会消耗大量 token，响应也慢。更好的方式是：

- 固定内容预生成：讲课脚本、提示阶梯、作业讲评模板、常见错误讲解、角色台词。
- 用户动态内容实时生成：举手提问、开放式追问、低置信度答案讲评。
- 热门内容进 Redis 缓存。
- 题库变化时用 RabbitMQ 异步重新生成 AI 教学材料。
- 用户学习时优先读取 MySQL / Redis 中的预生成内容。

所以这份方案会同时考虑产品效果和工程落地成本。

## 2. 方案对比

### 方案 A：老师讲课中间穿插提问

做法：

AI 老师不是一直讲，而是在关键知识点后插入小问题。例如：

- “刚才讲了 ref 的适用场景，那 number 类型为什么更适合用 ref？”
- “reactive 解构后可能会发生什么？”
- “你觉得 computed 和 watch 最大区别是什么？”

用户回答后立即得到反馈：

- 答对：老师夸奖，并补充更标准的表达。
- 答错：老师指出误区，用更简单的话再讲一遍。
- 不会：允许点击“我不会”，AI 给分层提示。

评估：

| 维度 | 结论 |
| --- | --- |
| 实现难度 | 低到中 |
| 可行性 | 很高 |
| 学习价值 | 很高 |
| 游戏感 | 中 |
| token 成本 | 低，可预生成 |
| 推荐程度 | 强烈推荐 |

优点：

- 符合学习规律，能触发主动回忆。
- 和题库天然结合，题目固定时可以提前生成课堂提问。
- 不需要复杂角色系统，第一版就能上线。
- 对所有用户都比较自然，不容易跑偏。

风险：

- 问得太频繁会打断课堂节奏。
- 问题太难会让用户挫败。
- 如果每个问题都调 AI 实时判断，响应会变慢。

解决方式：

- 每 2 到 3 个知识点插入一次提问，不要每句话都问。
- 题目难度从选择题、判断题、填空题，再到简答题逐渐升级。
- 先用关键词和选项规则本地评分，低置信度才调用 AI。

### 方案 B：课间场景，AI 同学向用户请教

做法：

在老师讲完一段后，进入一个“课间互助”场景。另一个 AI 同学向用户请教问题。

例如：

> AI 同桌：我刚才有点没听懂，为什么 reactive 解构后会丢响应式呀？你能给我讲一下吗？

用户回答后：

- 答对：同桌表达感谢和认可，增加好感度 / 羁绊值。
- 答错：同桌安慰用户，并一起请老师补讲。
- 不会：同桌说“那我们一起问老师吧”，转入老师讲解。

评估：

| 维度 | 结论 |
| --- | --- |
| 实现难度 | 中到高 |
| 可行性 | 中高 |
| 学习价值 | 中高 |
| 游戏感 | 很高 |
| token 成本 | 中，可通过预生成降低 |
| 推荐程度 | 建议作为增强玩法 |

优点：

- 情绪价值更强，用户更容易继续学。
- “教别人”会倒逼用户组织语言，学习效果好。
- 比单纯老师问答更有陪伴感。
- 适合做连续学习、打卡、角色成长。

风险：

- “美少女同学 + 好感度”容易让产品从学习偏向恋爱养成，影响主目标。
- 如果用户年龄不确定，强恋爱化设计可能有合规和舆论风险。
- 角色台词写不好会显得尴尬或低幼。
- 好感度如果和学习强绑定，用户可能为了刷数值而不是认真学习。

建议调整：

- 不建议第一版直接做强“美少女恋爱向”。
- 可以先做“AI 同桌 / 学习伙伴”，角色可选：学霸同桌、迷糊同桌、面试官同伴、项目组队友。
- “好感度”建议命名为“羁绊值 / 信任值 / 协作值”，弱化恋爱暗示。
- 夸赞和安慰要围绕学习表现，不做暧昧表达。

### 方案 C：老师课堂提问 + AI 同桌课间互助混合

做法：

课堂主线由 AI 老师掌控：

1. AI 老师讲 1 到 2 个知识点。
2. AI 老师插入一次小问题，检测用户是否理解。
3. 用户答完后获得即时反馈。
4. 每完成一小节，进入一次“课间互助”。
5. AI 同桌向用户请教一个问题。
6. 用户回答后增加羁绊值，并获得补充讲解。
7. 讲完整节课后进入作业。
8. 作业阶段仍然保持“提交一题，AI 讲评一题”。

评估：

| 维度 | 结论 |
| --- | --- |
| 实现难度 | 中 |
| 可行性 | 高 |
| 学习价值 | 很高 |
| 游戏感 | 高 |
| token 成本 | 可控 |
| 推荐程度 | 最推荐 |

优点：

- 老师提问保证学习效果。
- 同桌互动提高陪伴感和积极性。
- 两种互动有分工，不会全靠角色噱头。
- 题库固定时，大部分内容都能提前生成。

风险：

- 流程可能变长，用户想快速刷题时会觉得慢。
- 互动节点太多会稀释主线。
- 同桌系统如果做得太重，第一版开发成本会升高。

解决方式：

- 提供两种模式：快速学习、沉浸学习。
- 快速学习只保留老师提问和作业讲评。
- 沉浸学习开启 AI 同桌、羁绊值、课间互助。
- 每节课最多 1 到 2 次同桌互动，不要过密。

## 3. 推荐方案

推荐采用方案 C，但要控制范围：

**第一版做“老师课堂提问 + 轻量 AI 同桌课间互助”。**

不要一开始就做复杂恋爱养成，也不要让角色系统盖过学习系统。

产品定位应该是：

> AI 老师负责讲清楚，AI 同桌负责让用户愿意继续学。

推荐优先级：

1. 必做：老师讲课中间插入理解检测题。
2. 必做：答对 / 答错 / 不会 三种反馈。
3. 必做：讲完后布置作业，提交一题讲评一题。
4. 可做：AI 同桌课间请教用户。
5. 可做：羁绊值、夸赞、安慰。
6. 暂缓：复杂角色养成、礼物系统、剧情分支、恋爱化表达。

## 8. 可能出现的问题

### 8.1 用户觉得流程太慢

原因：

用户只想刷题，却被迫看剧情和互动。

解决：

- 提供“快速刷题”和“沉浸学习”两种模式。
- 快速刷题关闭 AI 同桌，只保留老师提示和逐题讲评。
- 沉浸学习开启课堂提问、课间互助、羁绊值。

### 8.2 游戏化盖过学习

原因：

用户为了刷羁绊值乱答、重复答、跳过理解。

解决：

- 羁绊值只作为轻奖励，不作为核心成长系统。
- 主成长指标仍然是掌握度、正确率、薄弱点减少。
- 答错也给鼓励，但不能给过高奖励。

### 8.3 角色设定引发争议

原因：

“美少女同学 + 好感度”可能被理解为恋爱化，尤其教育产品用户年龄不确定。

解决：

- 第一版用“AI 同桌 / 学习伙伴 / 协作值”。
- 角色可选，不强制。
- 台词保持鼓励、感谢、互助，不做暧昧表达。
- 后续如果明确面向成年用户，再考虑更强角色化皮肤。

### 8.4 AI 回答不稳定

原因：

实时 AI 可能答偏、答长、答得不像老师。

解决：

- 固定题库内容全部预生成并审核。
- 实时回答只允许使用当前课程上下文。
- Prompt 中限制输出长度、语气、结构。
- 对 AI 输出做敏感词和格式校验。

### 8.5 成本上升

原因：

同桌台词、老师台词、讲评内容都实时生成。

解决：

- 课堂节点全部预生成。
- 同桌常用反馈模板化。
- 作业讲评先规则评分。
- 热门课程进 Redis。
- 只有开放问题和低置信度答案调 AI。

## 9. 分阶段落地建议

### 第一阶段：学习效果优先

目标：

把 AI 老师讲课中间提问做出来。

功能：

- 课程脚本节点化。
- lecture 节点：老师讲课。
- checkpoint 节点：老师提问。
- 本地关键词评分。
- 答对 / 答错 / 不会反馈。
- 讲完后布置作业。

### 第二阶段：轻游戏化

目标：

加入 AI 同桌，提高陪伴感。

功能：

- classmate 节点。
- AI 同桌请教用户。
- 羁绊值 / 协作值。
- 答对夸赞，答错安慰。
- 老师补充讲解。

### 第三阶段：个性化推荐

目标：

让学习记录真正影响下一次学习。

功能：

- 统计薄弱关键词。
- 推荐下一节课程。
- 根据错题生成复习任务。
- 对高频错点重新生成补讲材料。

## 10. 最终结论

最推荐方案：

**老师课堂提问为主，AI 同桌课间互助为辅。**

原因：

- 课堂提问直接提高学习效果，开发成本低，最值得先做。
- AI 同桌能提升情绪价值和持续学习动力，但不应一开始做太重。
- 固定题库可以通过预生成和缓存解决 token 成本与响应速度问题。
- 混合方案能同时兼顾“学得会”和“愿意学”。

一句话产品定位：

> AI 老师负责把知识讲明白，AI 同桌负责让学习过程更有陪伴感，题库作业负责把掌握度沉淀成数据。
